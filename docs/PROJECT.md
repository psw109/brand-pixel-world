> **문서 목적:** 웹 기반 픽셀 마을 탐험 + 브랜드 건물 상호작용 서비스의 전체 그림을 한곳에 두고, 개발·기획이 바뀔 때마다 이 파일을 수정하며 맞춰 나간다.  
> **개정:** 상단 표에 날짜와 변경 요지만 적는다.

## 1. 프로젝트 한 줄 정의

**2D 탑다운 픽셀 마을을 드래그(또는 터치)로 둘러보고, 입점 가능 부지를 표시하며, 부지 또는 건물을 탭해 입점 문의·브랜드 경험(경량 콘텐츠·임베드·외부 링크 등)으로 이어지는 웹 서비스.**

- 설치 없이 브라우저에서 동작 (웹 + 모바일 브라우저 고려).
- “광고 플랫폼”으로 확장할 여지는 두되, **현 단계 범위**는 아래 [§3 범위](#3-현-단계-범위와-확장-여지)에 따른다.

## 2. 이 프로젝트가 하는 일 (사용자 관점)

| 단계 | 사용자 행동 | 기대 결과 |
|------|-------------|-----------|
| 진입 | URL로 접속 | 픽셀 마을(월드)이 보임 |
| 탐색 | 마우스 드래그 / 터치 드래그 (필요 시 줌은 추후) | 마을을 이동하며 **부지(입점 가능 구역)**·건물·배경 탐색 |
| 상호작용 (부지) | **비어 있는 입점 부지** 탭 | 오버레이에서 입점·홍보 문의: `mailto:` 링크, 또는 웹 폼 제출(POST) 등 — 제출 내용은 **관리자 페이지**에서 확인 |
| 상호작용 (건물) | 건물이 점유한 **타일(한 칸 이상)** 탭 | **그 건물에만 정의된** 다음 중 하나 또는 조합: 경량 창(애니 등), iframe, 외부 페이지 이동, 직접 만든 내부 페이지 |

### 2.1 부지(Building lot) 개념

- **부지**는 “아직 건물이 없지만, 여기에 브랜드 건물이 들어올 수 있다”는 것을 **마을 위에서 시각적으로** 보여 주는 오브젝트/타일/표시다 (예: 빈 터, “FOR LEASE” 느낌의 장식, 반투명 마커 등 — 그래픽은 구현 시 결정).
- 부지와 건물이 차지하는 **면적은 고정이 아닐 수 있다** (예: 2칸짜리 부지, 4칸짜리 부지·건물). 데이터는 **점유 타일 집합(footprint)** 으로 표현하는 방향을 [§5.3](#section-5-3-world-model)에서 다룬다.
- 부지는 **건물이 들어오기 전** 단계에서만 의미가 있고, 해당 부지에 건물이 배치되면 동일 **footprint** 는 **건물 탭** 동작으로 전환된다 (또는 부지 레이어를 숨기고 건물만 남긴다). **건물 교체**는 같은 부지 ID·다른 건물 레코드로 이어지도록 두면 이력·운영이 분리하기 쉽다.
- 부지 자동생성 알고리즘 관련 학습 필요 !!!

#### 탭 입력 (멀티 타일)

- 멀티 타일일 때는 **어느 타일을 눌러도** 같은 부지/건물로 인식하거나, **대표 타일(앵커)**만 히트로 잡는 등 규칙을 구현 시 정한다.

### 2.2 입점 문의·접수 채널 (클라이언트 → 운영)

| 방식 | 설명 | 비고 |
|------|------|------|
| **메일** | 부지 탭 후 `mailto:`로 미리 채운 제목·본문(부지 ID 등)으로 발송 | 구현 단순, 스팸·추적은 약함 |
| **웹 폼 + POST** | 같은 오버레이에서 이름·연락처·메모·부지 ID를 입력 → API/Supabase 등으로 저장 | 관리자 페이지와 짝지우기 좋음 |
| **관리자 확인** | 접수 목록·상태(신규/연락함/계약 등)를 **관리자 전용 페이지**에서 조회 | 인증·권한 설계 필요 |

초기 프로토타입은 **메일만** 또는 **폼 + 간단 저장** 중 하나로 시작하고, 나머지는 확장해도 된다.

**건물별 예시 (기획 확정 예시)**

| 구분 | 예시 | 설명 |
|------|------|------|
| **A** | 유튜브 임베드 | 탭 시 오버레이에 YouTube iframe(또는 공식 임베드 패턴)으로 영상 노출(저작권 체크) |
| **B** | 쇼핑몰로 이동 | 탭 시 새 탭/같은 탭에서 쇼핑몰 URL로 이동하거나, iframe으로 일부만 삽입 |
| **C** | 자체 개발 페이지 | 팀이 만든 React 등 페이지를 라우트로 두고, 건물 탭 시 그 페이지를 오버레이 또는 전환으로 표시 |

## 3. 현 단계 범위와 확장 여지

### 3.1 이번에 넣을 것 (핵심)

- 마을 월드 표시 + **드래그(패닝)** 로 시야 이동 (웹: 마우스 / 모바일: 터치).
- **입점 부지** 표시(시각적 구분) 및 **부지 탭** 시 입점·홍보 문의 UI (메일 및/또는 폼 POST).
- (선택) **관리자 페이지**에서 문의 접수 목록 확인 — 백엔드·인증이 생기는 시점에 맞춰 범위 조정.
- 건물 배치(최소 1개 이상) 및 **건물 탭 감지**.
- 건물마다 다른 **행동 타입** (임베드 / 외부 링크 / 내부 페이지 / 조합).
- 탭 시 **오버레이 UI**(월드 위에 덮이는 화면): 창·iframe·전환·문의 폼 등 -> 아마도 새로운 페이지로 이동(?).

### 3.2 나중에 붙여도 되는 것 (문서에만 적어 두고 우선순위는 개발 중 조정)

- 줌, 무한 청크, 건물 클릭 로그·광고주 대시보드, 광고주 셀프 CMS, 실시간 다국어 등.

## 4. 화면 구조: 레이어(스택)

건물의 “층”이 아니라, **화면에 그려지는 순서(z-order, 아래→위)** 를 나눈 개념이다. 개발·디자인 시 역할을 분리하기 위한 용어이며, 물리적으로 페이지가 반드시 두 개일 필요는 없다.

| 겹침 순서 | 레이어 이름 | 역할 |
|-----------|-------------|------|
| **아래** (먼저 그려짐) | **월드 레이어** | 픽셀 마을(타일/스프라이트). 드래그로 카메라 이동. **부지·건물** 탭 입력을 받음 (같은 **footprint** 에 부지와 건물이 동시에 존재하지 않도록 설계). |
| **위** (나중에 그려짐) | **오버레이 레이어** | 부지/건물 탭 후 뜨는 UI 전부: 입점 문의(메일·폼), 모달, 바텀시트, iframe, 자체 페이지 콘텐츠(레이어X). |

**용어 정리:** “레이어”는 포토샵·피그마·게임 엔진에서 쓰는 것과 같이 **투명도 있는 두 장을 위에 포개는 것**에 가깝다. **스택**은 같은 맥락에서 “아래/위”만 구분할 때 쓴다.

**주의:** 모바일에서 스크롤·뒤로가기·터치가 **월드와 오버레이**에 동시에 먹히지 않도록, 오버레이가 열릴 때 포커스·스크롤 잠금 정책을 정한다 (구현 단계에서 상세화).

## 5. 건물(브랜드 오브젝트) 설정 — 데이터 관점

건물마다 동작이 다르므로, **건물 하나 = 설정 한 줄(또는 JSON 객체 하나)** 로 두면 이후 DB·JSON·CMS로 옮기기 쉽다.

### 5.1 권장 필드 (초안)

필요에 따라 행을 추가·삭제하며 이 표를 업데이트한다.

| 필드 | 설명 | 예시 |
|------|------|------|
| `id` | 건물 고유 식별자 | `"building_cafe_01"` |
| `label` | 표시용 이름 (디버그·툴팁) | `"샘플 카페"` |
| `footprint` | 점유 타일(면적) | [§5.3](#section-5-3-world-model) 참고. 단일 칸이면 1×1로 표현 가능 |
| `sprite` | 건물 그래픽 참조 | 에셋 키 또는 URL |
| `interaction.type` | 탭 시 동작 유형 | 아래 [§5.2](#52-interactiontype-초안-값) 참고 |
| `interaction.payload` | 유형별 데이터 | URL, videoId, 내부 라우트 경로, iframe src 등 |
| `interaction.openIn` | 링크·외부 콘텐츠 열기 방식 | `overlay` / `same_tab` / `new_tab` (필요 시 확장) |

### 5.2 `interaction.type` 초안 값

| 값 | 의미 |
|----|------|
| `youtube_embed` | 유튜브 임베드 (iframe; `payload`에 video ID 또는 embed URL 규칙 합의) |
| `iframe` | 범용 iframe (`payload.src`) |
| `external_link` | 외부 URL로 이동 (쇼핑몰 등) |
| `internal_route` | 우리가 만든 페이지(React 라우트 등) |
| `lightweight_overlay` | 짧은 애니메이션/Lottie/HTML만 있는 경량 창 (`payload`에 리소스 참조) |
| `composite` | 순서 정의 필요: 예) 경량 재생 후 링크 버튼 — 나중에 `steps[]` 등으로 확장 |

실제 프로젝트에서는 타입을 줄이고 `payload`로만 구분하는 것도 가능하다. **팀이 합의한 뒤 이 문서 표를 수정한다.**

### 5.3 월드 그리드·부지·건물 데이터 모델 (권장)

<a id="section-5-3-world-model"></a>

#### 5.3.1 확장을 고려한 방향

- **맵(타일 그리드)** 과 **배치(부지·건물)** 를 분리하는 것을 권장한다.  
  - 바닥·장식 등은 **타일맵** 레이어로 두고,  
  - “입점 가능 부지”“건물”은 각각 **엔티티 + footprint** 로 둔다.
- **가변 면적**(부지 2칸·건물 4칸 등)과 **건물 교체·이력**까지 염두에 두면, **한 좌표 = 한 행**인 `status: empty | building` 단일 슬롯 테이블만으로는 표현이 금방 지저분해진다.  
  - 단일 슬롯 방식은 **모든 사물이 1칸**일 때 단순·유리하다.
- **권장:** **부지(Lot)** 와 **건물(Building)** 을 별도 엔티티로 두고, 둘 다 **동일한 footprint 규칙**으로 점유 타일을 기술한다. 입점 후에는 해당 부지에 연결된 건물 레코드로 전환하거나, `lot_id` + `building_id`로 연결해 **교체 시 건물만 갈아끼우기** 쉽게 한다.

#### 5.3.2 Footprint(점유 면적) 표현 — 선택지

| 방식 | 설명 | 장단 |
|------|------|------|
| **타일 ID 목록** | `[(x,y), (x+1,y), …]` | L자 등 불규칙 형태 가능하나 정사각형/직사각형을 . 저장량은 타일 수에 비례. |
| **앵커 + 가로×세로** | 좌상단 + `w×h` | 직사각형만. 단순하고 쿼리·에디터에 편함. |
| **폴리곤/비트마스크** | 복잡한 형태 | 필요 시 도입. 초기 비용↑ |

프로토타입은 **직사각형(앵커 + w×h)** 으로 시작하고, 불규칙 형태가 필요해지면 **타일 목록**으로 확장하는 식이 무난하다.

#### 5.3.3 부지(Lot) 필드 — 초안

| 필드 | 설명 | 예시 |
|------|------|------|
| `id` | 부지 고유 식별자 (문의에 포함) | `"lot_plaza_03"` |
| `footprint` | 점유 타일(면적) | §5.3.2 방식 중 하나 |
| `display` | 시각 스타일 | 스프라이트 키, “빈 땅” 에셋 등 |
| `inquiry` | 탭 시 동작 | `mailto_template` / `form_post` / 둘 다 |
| `inquiry.endpoint` | 폼 제출 시 API 경로 등 | 구현 후 기입 |
| `inquiry.mailto` | 메일 안내 시 | 제목·본문 템플릿에 `{{lot_id}}` 치환 |

**충돌 규칙:** 서로 다른 부지·건물의 **footprint 가 겹치지 않도록** 검증한다. (같은 격자 칸에서 부지와 건물이 동시에 존재하지 않음.)

#### 5.3.4 건물(Building)과 footprint

`§5.1`의 위치 필드는 **`footprint`** 로 통일하는 것을 권장한다. 렌더·히트는 스프라이트 앵커에 맞춰 `footprint` 와 별도로 둘 수 있으나, **데이터 상의 점유 범위**는 footprint로 단일화하는 편이 충돌 검사에 유리하다.

### 5.4 입점 문의(Lead) 저장 — 개념 (초안)

웹 폼을 쓸 경우 최소 필드 예시: `lot_id`, `contact_email`, `message`, `created_at`, `status`(관리자용). 저장소는 Supabase 테이블 등으로 두고, **관리자 페이지**는 인증된 역할만 접근한다.

## 6. 기술 스택 (현재 확정안)

현재 단계에서는 아래 스택/패턴으로 개발을 진행한다. 변경 시 이 표와 관련 문서를 함께 갱신한다.

| 영역 | 선택 | 비고 |
|------|------|------|
| 프론트 앱 기반 | **Next.js + TypeScript** | 1차 구현 기본. JavaScript보다 TypeScript를 우선 사용해 도메인 타입(`footprint`, `interaction`)을 강제한다 |
| 월드·입력·픽셀 렌더링 | **Phaser 3** | 월드는 캔버스 기반으로 구현하고 패닝/타일/스프라이트를 담당한다 |
| UI·오버레이·라우팅 | **React (Next App Router)** | 오버레이는 DOM 기반으로 구현하고 iframe·모달·내부 페이지 라우팅을 담당한다 |
| 상태 | **Zustand** 등 가벼운 store | 월드 카메라 / 열린 부지·건물 ID / 오버레이 여부 |
| 백엔드·저장 (선택) | **Supabase + Next API Routes(Serverless)** | 별도 상시 백엔드 서버 없이 문의(lead) 저장·조회부터 시작 가능 |
| 배포 | **Vercel** | Next.js + Serverless/API Routes 배포에 적합 |
| 관리자 UI | React 라우트 `/admin` 등 + 인증 | 문의 목록·상태 변경 (범위는 단계적으로) |

```
  (선택) ADR/     ← 중요한 결정만 짧게
```

- **큰 방향이 바뀌면** §2·§5·§6을 먼저 고친다. 부지·문의·footprint는 §2.1–2.2, §5.3–5.4를 본다.

## 8. 용어 짧게

| 용어 | 의미 |
|------|------|
| 월드 / 마을 | 사용자가 드래그로 돌아다니는 2D 맵 전체 |
| 타일맵 / 그리드 | 바닥·장식 등 맵의 격자 레이어 (배치 엔티티와 분리) |
| footprint | 부지 또는 건물이 차지하는 타일 집합(면적); 2칸·4칸 등 가변 |
| 부지 / 입점 슬롯 | 건물이 들어오기 전에 표시되는 “입점 가능” 구역; 탭 시 입점 문의로 연결 |
| 건물 | 탭 가능한 브랜드 오브젝트; 행동은 설정으로 결정 |
| 오버레이 | 월드 레이어 **위**에 포개지는 UI 레이어(모달·폼·iframe 등). 건물 층과 무관 |
| 기믹 | 넓은 의미로 사용: 미니게임뿐 아니라 **경량 연출·임베드·링크** 포함 |
| 입점 문의 / Lead | 부지 탭 후 메일 또는 폼으로 들어오는 클라이언트 요청; 관리자가 확인 |

## 9. 구현 우선순위 (초안)

인프라 연결은 [`SETUP.md`](./SETUP.md) / 팀원 실행은 [`ONBOARDING.md`](./ONBOARDING.md). 아래는 제품 구현 순서 제안이다.

1. 월드(Phaser) 캔버스 + 오버레이(React) 기본 골격
2. `lots` / `buildings` 데이터를 Supabase에서 읽기(anon SELECT 정책)
3. 문의 폼 → API → Supabase `leads` 저장
4. `dev`에서 QA 후 `main` 반영
